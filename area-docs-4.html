Introduction

Area is a powerful AI platform that gives you access to state-of-the-art language models through a simple, flexible API. Build intelligent applications, automate workflows, and create new experiences at scale.

📖 15 min readUpdated April 2025

What is Area?

Area provides developer access to AI models through a REST API. Whether you're building a chatbot, automating document processing, writing code assistants, or creating any AI-powered product, Area gives you the building blocks to do it reliably and at scale.

Our models are trained with an emphasis on being helpful, harmless, and honest — making them well-suited for real-world deployment where safety and reliability are non-negotiable.

💡
New to Area? Start with the Quickstart guide to make your first API call in under 5 minutes.

Core Capabilities

Text Generation

Send a prompt, get an intelligent response. The models understand context, follow instructions, maintain coherent conversations over many turns, and produce output in virtually any format — prose, code, JSON, markdown, or structured data.

Vision & Multimodal Input

All Area models accept image inputs alongside text. Send screenshots, photos, diagrams, documents, and charts — the model reads them natively. PDFs up to 32 MB and 100 pages are supported.

Tool Use (Function Calling)

Define tools with typed parameters and the model decides when to call them. Enables AI agents that can search the web, query databases, call APIs, write and execute code, and more. Fully composable for multi-step agentic workflows.

Streaming

Stream responses token-by-token for real-time user experiences using server-sent events (SSE). First-token latency under 300ms on most requests.

Extended Thinking

Enable extended thinking on Opus and Sonnet to have the model reason step-by-step before answering. Significantly improves performance on math, coding, logic, and multi-step reasoning tasks.

Prompt Caching

Cache frequently-used portions of prompts and save up to 90% on input token costs. Cache lifetime is 5 minutes with automatic refresh on use.

API Design Principles

  • REST-based: Standard HTTP verbs, JSON request and response bodies, stateless architecture.
  • Versioned: API versioned via a required header so we can evolve without breaking integrations.
  • Idiomatic SDKs: Official Python and TypeScript SDKs with full type safety, streaming helpers, and retry logic.
  • Predictable errors: Structured error responses with machine-readable codes and actionable guidance.
  • Generous limits: High default rate limits, per-token pricing with no minimums, usage tiers that grow with you.

Step 5 — Multi-Turn Conversation

python
history = []

def chat(user_msg):
    history.append({"role": "user", "content": user_msg})
    resp = client.messages.create(
        model="area-sonnet-3-5-20241022",
        max_tokens=1024,
        messages=history
    )
    text = resp.content[0].text
    history.append({"role": "assistant", "content": text})
    return text

print(chat("What is the capital of France?"))
print(chat("What is the population of that city?"))
print(chat("Name 3 famous landmarks there"))

Messages API
Core Concepts

Messages API

The primary interface for interacting with Area models. Send a structured conversation, receive a structured response. Everything in Area flows through this single endpoint.

📖 12 min read

Endpoint

POSThttps://api.area.ai/v1/messages

Request Parameters

ParameterTypeDescription
modelRequiredstringThe model ID. Example: area-sonnet-3-5-20241022. See Models page for all IDs.
messagesRequiredarrayArray of message objects. Each has role ("user" or "assistant") and content. Must alternate user/assistant and end with a user message.
max_tokensRequiredintegerMax tokens to generate. Varies by model. Too low may truncate responses.
systemOptionalstringSystem prompt defining the assistant's role, constraints, and context. Persists for the entire conversation.
temperatureOptionalnumberSampling temperature 0–1. Higher = more random. Default: 1.0. Use 0 for deterministic output.
top_pOptionalnumberNucleus sampling. Sample from tokens comprising the top P probability mass. Range: 0–1.
top_kOptionalintegerOnly sample from the top K tokens. Reduces low-probability long-tail responses.
stop_sequencesOptionalarrayUp to 8192 custom sequences that halt generation when encountered. Not included in the response.
streamOptionalbooleanStream responses via server-sent events. Response body becomes a stream of events instead of a single JSON object.
toolsOptionalarrayTool definitions the model may call. Each tool has name, description, and input_schema (JSON Schema).
tool_choiceOptionalobject{"type":"auto"} (default), {"type":"any"} (must use a tool), or {"type":"tool","name":"fn"} (specific tool).
metadataOptionalobjectOptional metadata. Supports user_id (string, max 256 chars) for abuse detection and monitoring.

Response Object

json
{
  "id": "msg_01XFDUDYJgAACzvnptvVoYEL",
  "type": "message",
  "role": "assistant",
  "content": [
    { "type": "text", "text": "The response text..." }
  ],
  "model": "area-sonnet-3-5-20241022",
  "stop_reason": "end_turn",
  "stop_sequence": null,
  "usage": {
    "input_tokens": 25,
    "output_tokens": 189
  }
}

stop_reason Values

ValueMeaning
end_turnModel completed its response naturally
max_tokensHit max_tokens limit — increase it or use continuation
stop_sequenceA custom stop sequence was encountered
tool_useModel is requesting a tool call — provide results and continue

Choosing a Model

Area Opus — Maximum Intelligence

Use Opus when accuracy and capability matter most: complex multi-step reasoning, nuanced writing, code generation with deep logic, research synthesis, and tasks where the cost of errors is high. Consistently achieves the best results across all benchmarks.

Best for: Complex analysis, PhD-level reasoning, long-form research, high-stakes code generation, nuanced creative work.

Area Sonnet — Optimal for Production

Sonnet delivers excellent performance at 5× lower cost than Opus. Most production workloads achieve nearly Opus-level quality at a fraction of the price. The latest Sonnet (3.5) outperforms Opus 3 on many coding benchmarks.

Best for: Customer support agents, code review, data analysis, content generation, RAG pipelines, agentic workflows.

Area's fastest and most affordable model. Sub-second latency for most requests. Haiku 3.5 is significantly more capable than Haiku 3 and includes full vision and tool use support.

Best for: High-volume classification, simple Q&A, entity extraction, real-time autocomplete, latency-critical features.

Streaming
Core Concepts

Streaming

Stream responses token-by-token using server-sent events. First tokens arrive in under 300ms, making your app feel instant even for long outputs.

Why Stream?

Without streaming, users wait for the entire response — which can take seconds for long outputs. With streaming, text appears immediately as it is generated, dramatically improving perceived performance and user experience.

Streaming is especially critical for: chatbots, code generation, long-form writing, interactive assistants, and any feature where latency is visible to the user.

Python — Stream Context Manager

python
import area

client = area.Area()

# Recommended: streaming context manager
with client.messages.stream(
    model="area-sonnet-3-5-20241022",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a poem about the ocean"}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

# Access the complete message after streaming ends
final = stream.get_final_message()
print(f"
Tokens: {final.usage.input_tokens} in, {final.usage.output_tokens} out")

TypeScript — Async Iterator

typescript
import Area from "@area-ai/sdk";

const client = new Area();

const stream = await client.messages.stream({
  model: "area-sonnet-3-5-20241022",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Write a poem about the ocean" }],
});

// Option 1: stream text only
stream.on("text", (text) => process.stdout.write(text));

// Option 2: iterate all events
for await (const event of stream) {
  if (event.type === "content_block_delta" &&
      event.delta.type === "text_delta") {
    process.stdout.write(event.delta.text);
  }
}

const final = await stream.finalMessage();
console.log(final.usage);

Raw SSE Events

EventDescription
message_startBeginning of response. Contains message ID and initial usage.
content_block_startA new content block is starting (text or tool_use).
content_block_deltaA chunk of content. For text: {"type":"text_delta","text":"..."}.
content_block_stopCurrent content block has ended.
message_deltaUpdates to top-level message fields (stop_reason, usage).
message_stopMessage is complete. Final event.
errorError occurred. Stream terminates after this.

Streaming in a Web App (Next.js)

typescript
// app/api/chat/route.ts
import Area from "@area-ai/sdk";

export const runtime = "edge";

const area = new Area();

export async function POST(req: Request) {
  const { messages } = await req.json();

  const stream = await area.messages.stream({
    model: "area-sonnet-3-5-20241022",
    max_tokens: 2048,
    messages,
  });

  return new Response(stream.toReadableStream());
}

Tool Use
Core Concepts

Tool Use

Give Area models the ability to call your functions. Define tools with typed parameters and the model decides when and how to use them — enabling powerful AI agents that interact with real systems.

How Tool Use Works

1
Define tools and send request
Include tool definitions in the tools parameter. The model evaluates the message and decides whether to call a tool.
2
Model returns tool_use block
If calling a tool, the model responds with stop_reason: "tool_use" and a tool_use block containing the tool name and typed input.
3
Execute tool in your code
Your application runs the tool with the provided input. Area never executes code — tool execution always happens on your side.
4
Send results back
Append a tool_result block and make another API call. The model uses the result to generate a final natural-language response.

Defining Tools

python
tools = [
  {
    "name": "get_weather",
    "description": "Get current weather for a city. Use when user asks about weather or temperature.",
    "input_schema": {
      "type": "object",
      "properties": {
        "location": {
          "type": "string",
          "description": "City and country, e.g. Mumbai, India"
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "description": "Temperature unit. Default: celsius"
        }
      },
      "required": ["location"]
    }
  },
  {
    "name": "search_database",
    "description": "Search the product database for items matching a query.",
    "input_schema": {
      "type": "object",
      "properties": {
        "query": {"type": "string", "description": "Search query"},
        "limit": {"type": "integer", "description": "Max results to return", "default": 10}
      },
      "required": ["query"]
    }
  }
]

Full Tool Use Loop

python
import json, area

client = area.Area()

def run_tool(name, inputs):
    if name == "get_weather":
        return {"temp": 32, "unit": "C", "condition": "sunny"}
    elif name == "search_database":
        return [{"id": 1, "name": "Widget Pro", "price": 49.99}]

messages = [{"role": "user", "content": "What's the weather in Mumbai?"}]

while True:
    resp = client.messages.create(
        model="area-sonnet-3-5-20241022",
        max_tokens=1024,
        tools=tools,
        messages=messages
    )

    # Done — model gave a final text response
    if resp.stop_reason != "tool_use":
        print(resp.content[0].text)
        break

    # Process all tool calls
    messages.append({"role": "assistant", "content": resp.content})
    tool_results = []

    for block in resp.content:
        if block.type == "tool_use":
            result = run_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": json.dumps(result)
            })

    messages.append({"role": "user", "content": tool_results})

Tool Choice Options

ValueBehavior
{"type": "auto"}Model decides whether to use a tool (default)
{"type": "any"}Model must use at least one tool
{"type": "tool", "name": "fn_name"}Model must call this specific tool
💡
Write great tool descriptions. The model uses your description to decide when to call a tool. Be explicit about when it should and should not be used. Include edge cases and examples in the description.

Prompt Caching
Capabilities

Prompt Caching

Cache reused portions of your prompts and save up to 90% on input token costs. Ideal for large system prompts, reference documents, and tool definitions.

How It Works

Add "cache_control": {"type": "ephemeral"} to any content block. Area stores that prefix in a cache after first use. Subsequent requests reading from cache cost 90% less.

MetricValue
Cache lifetime5 minutes (extended on each use)
Write cost1.25× base input price
Read cost0.10× base input price (90% savings)
Min cacheable tokens1024 (Sonnet/Opus), 2048 (Haiku)
Max breakpoints4 per request

Basic Usage

python
BIG_SYSTEM_PROMPT = "[5000+ token knowledge base, guidelines, examples...]"

resp = client.messages.create(
    model="area-sonnet-3-5-20241022",
    max_tokens=1024,
    system=[{
        "type": "text",
        "text": BIG_SYSTEM_PROMPT,
        "cache_control": {"type": "ephemeral"}  # cache this!
    }],
    messages=[{"role": "user", "content": "Hello"}]
)

print(resp.usage.cache_creation_input_tokens)  # written to cache
print(resp.usage.cache_read_input_tokens)     # read from cache

When to Cache

  • Large system prompts: Product knowledge bases, style guides over 2K tokens
  • RAG context: Retrieved document chunks included in every request
  • Few-shot examples: Long input/output example lists
  • Tool definitions: When you have many detailed tool descriptions
  • Conversation history: Cache the growing history in long conversations
💡
Cost math: A 10K token system prompt used 100×/minute costs ~87% less with caching — one write at 1.25× plus 99 reads at 0.10× vs 100 full reads at 1.0×.

Prompt Engineering
Best Practices

Prompt Engineering

Effective prompting unlocks the full capability of Area models. These techniques are battle-tested across thousands of production deployments.

Fundamental Principles

Be Specific and Explicit

Area models do exactly what you ask. Ambiguous instructions lead to inconsistent output. State your constraints, format requirements, and expectations clearly.

⚠️
Vague: "Summarize this."
Specific: "Summarize this article in 3 bullet points (max 20 words each) for a non-technical audience. Focus on business impact."

Use System Prompts for Persistent Behavior

The system prompt defines role, format, tone, and constraints that apply to every turn. Think of it as the model's permanent instructions.

text — example system prompt
You are a customer support agent for TechFlow, a SaaS project management tool.

## Your Role
Help users resolve issues, answer questions, and guide them through the product.
Be professional, empathetic, and solution-focused.

## Response Format
- Use short paragraphs (3-4 sentences max)
- Use numbered steps for multi-step instructions
- Always end with: "Is there anything else I can help with?"

## Constraints

Retry Logic

python
import time, random, area

def create_with_retry(client, max_retries=3, **kwargs):
    for attempt in range(max_retries):
        try:
            return client.messages.create(**kwargs)
        except area.RateLimitError:
            if attempt == max_retries - 1: raise
            wait = (2 ** attempt) + random.uniform(0, 1)
            print(f"Rate limited. Retrying in {wait:.1f}s...")
            time.sleep(wait)
        except (area.APIConnectionError, area.InternalServerError):
            if attempt == max_retries - 1: raise
            time.sleep(2 ** attempt)
        except area.BadRequestError:
            raise  # Don't retry client errors

Safety Guidelines

Safety Guidelines

Area models are trained to be helpful, harmless, and honest. Understanding their safety properties helps you build applications that are powerful and responsible.

Model Behavior

Area models are trained with Constitutional AI (CAI) and RLHF, instilling core values around safety, accuracy, and honesty that persist even under adversarial prompting. The models will decline requests that could cause significant harm, acknowledge uncertainty rather than fabricating facts, and maintain these behaviors regardless of instructions.

Usage Policy

The Area API may not be used to generate:

  • Content that sexually exploits minors
  • Detailed instructions for biological, chemical, nuclear, or radiological weapons
  • Content designed to facilitate real-world violence against specific people
  • Malware or cyberweapons intended to cause widespread damage

Full policy at area.ai/legal/usage-policy.

Sensitive Domains

For healthcare, legal, mental health, or financial advice applications:

  • Add explicit disclaimers in your system prompt: "You are not a licensed doctor. Always recommend consulting a professional."
  • Display appropriate disclaimers in your UI
  • Review model outputs before surfacing them in high-stakes contexts
  • Use metadata.user_id for abuse monitoring

Prompt Injection Defense

When passing user content to the model, protect against prompt injection attacks where users try to override your system instructions.

  • Wrap user content in XML tags: <user_input>...</user_input>
  • Reinforce constraints in the message: "Remember: only answer questions about our product."
  • Validate and sanitize user inputs before including in prompts
  • Log and monitor unusual outputs for human review

Core Concepts

System Prompts

Define persistent context, persona, and behavioral guidelines for the assistant.

Overview

The system parameter sets instructions that apply to every turn of the conversation. Unlike user messages, the system prompt is not shown to the model as coming from a person — it sets the foundational context from which the model operates.

Best Practices

  • Keep system prompts focused — one clear role and set of instructions
  • Use headings (##) to organize long system prompts
  • Be explicit about format, tone, and constraints
  • Include examples of good responses when output format matters
  • Cache long system prompts with Prompt Caching to reduce costs

Testing & Evals

Build confidence in your AI integration through systematic evaluation before shipping to production.

Why Evals Matter

Unlike traditional software where unit tests give exact pass/fail results, AI outputs are probabilistic. Evals give you a statistical measure of quality — and help you catch regressions when you change models or prompts.

Eval Workflow

1
Build a golden dataset
Collect 50-200 representative inputs with expected outputs or quality criteria. Include edge cases and examples from production.
2
Define scoring criteria
Exact match for classification tasks. LLM-as-judge for open-ended quality. Human review for high-stakes domains.
3
Run via Batch API
Submit your dataset using the Batch API for cost-efficient evaluation. 50% off standard pricing.
4
Analyze and iterate
Review failures, update your prompt, and re-run. Track scores over time as you make changes.
